## *Краткое руководство*

**Краткое содержание:*- Используйте `quick-sharun.sh` для сборки вашего приложения со всеми его зависимостями в действительно портативный AppImage, работающий на любой системе Linux.

Что нужно:

- Система Linux (желательно Arch Linux для сборки)
- Базовые знания написания скриптов оболочки
- Приложение, которое вы хотите упаковать (желательно установленное в /usr)

### *Базовый рабочий процесс*

Создание AppImage с помощью quick-sharun включает следующие шаги:

1. **Установите ваше приложение*- и его зависимости в вашей системе сборки.
2. **Загрузите quick-sharun.sh*- из этого [репозитория](../src/scripts/quick-sharun.sh).
3. **Установите переменные окружения*- для настройки `quick-sharun`.
4. **Запустите quick-sharun*- с путем к исполняемому файлу вашего приложения (и библиотекам) для развертывания.
5. **Сгенерируйте AppImage*- с флагом `--make-appimage`.

Вот и все! Скрипт выполнит следующие действия:

- Обнаружит и соберет все необходимые библиотеки (включая те, которые открываются посредством dlopen).
- Создаст портативный AppImage, который будет работать везде.

### *Пошаговый пример*

Давайте создадим AppImage для простого приложения. Вот минимальный пример:

```shell
#!/bin/sh
set -eux

ARCH="$(uname -m)"
SHARUN="https://raw.githubusercontent.com/pkgforge-dev/Anylinux-AppImages/refs/heads/main/useful-tools/quick-sharun.sh"

# Настройка AppImage
export ICON=/usr/share/icons/hicolor/256x256/apps/myapp.png
export DESKTOP=/usr/share/applications/myapp.desktop
export OUTPATH=./dist
export OUTNAME=myapp-"$ARCH".AppImage

# Установка приложения (пример с использованием pacman)
pacman -Syu --noconfirm base-devel wget myapp

# Загрузка и запуск quick-sharun
wget "$SHARUN" -O ./quick-sharun
chmod +x ./quick-sharun

# Сборка приложения
./quick-sharun /usr/bin/myapp

# Создание AppImage
./quick-sharun --make-appimage
```

**Использование уменьшенных пакетов**:

```shell
EXTRA_PACKAGES="https://raw.githubusercontent.com/pkgforge-dev/Anylinux-AppImages/refs/heads/main/useful-tools/get-debloated-pkgs.sh"

wget "$EXTRA_PACKAGES" -O ./get-debloated-pkgs.sh
chmod +x ./get-debloated-pkgs.sh

# Устанавливает упрощенные пакеты MESA, Vulkan, Qt, GTK, libicudata и другие
./get-debloated-pkgs.sh --add-mesa --prefer-nano

# Некоторые приложения могут также потребовать эти команды
./get-debloated-pkgs.sh --add-common --prefer-nano ffmpeg-mini intel-media-driver-mini
```

### *Доступные переменные среды*

**Основные параметры конфигурации:**
- `APPDIR` - Куда создавать каталог приложения (по умолчанию: `$PWD/AppDir`)
- `ICON` - Путь к значку приложения
- `DESKTOP` - Путь к файлу .desktop
- `OUTPATH` - Куда сохранять AppImage (по умолчанию: `$PWD`)
- `OUTNAME` - Имя файлаx Вывод файла AppImage; если имя не указано в файле .desktop, будет использовано имя из этого файла.

### *Использование хуков*

Хуки — это скрипты, которые автоматически решают распространённые проблемы. Добавьте их, используя переменную `ADD_HOOKS`, каждый хук должен быть  отделён от предыдущего двоеточием:

```shell
export ADD_HOOKS="self-updater.bg.hook:fix-namespaces.hook"
```

Все хуки подхватываются сгенерированным `AppRun`. Более старые суффиксы `.bg.hook` и `.src.hook` сохраняются только для совместимости, поэтому в новых примерах следует использовать конечные разрешения только `.hook`. **Прочитать подробнее можно в*- [helpers.md](helpers.md)

**Параметры развертывания:**

- `DEPLOY_OPENGL=1` - Включает библиотеки OpenGL (MESA) (должно происходить автоматически)
- `DEPLOY_VULKAN=1` - Включает библиотеки Vulkan (MESA) (должно происходить автоматически)
- `DEPLOY_PYTHON=1` - Включает Python с использованием развертывания Python (отключено по умолчанию)
- `DEPLOY_LOCALE=1` - Развертывает файлы локалей (по умолчанию: включено)
- `ANYLINUX_LIB=1` - Предварительно загружает библиотеку, которая исправляет ряд распространенных проблем, затрагивающих AppImage (включено по умолчанию).
- `GTK_CLASS_FIX=1` - Включает в себя небольшой заглушечный модуль, исправляющий параметр WM_CLASS для приложений GTK (отключен по умолчанию).
- `OPTIMIZE_LAUNCH=1` - Ускоряет время запуска AppImage, используя образ профиля DWARFS (по умолчанию: отключено). Это очень похоже на оптимизации PGO в компиляторах. Часто включать это не требуется, поскольку DWARFS сам по себе во много раз быстрее, чем SquashFS. Во многих случаях время запуска практически идентично времени запуска нативных приложений (±300 мс на системе с процессором 2016 года).
- `STRACE_MODE=1` - Использует strace для поиска динамически загружаемых библиотек (включено по умолчанию)
- `STRIP=1` - Удаляет отладочные символы для уменьшения размера (включено по умолчанию, если не установлен NO_STRIP)
- `DEBLOAT_LOCALE=1` - Удаляет ненужные локали для уменьшения размера AppImage (включено по умолчанию)

### *Проблема*

Долгое время рекомендуемой практикой создания AppImages было включение большинства необходимых приложению библиотек, но не всех, таких как libc, динамический компоновщик и несколько других, упомянутых в [списке исключений](https://github.com/AppImageCommunity/pkg2appimage/blob/master/excludelist)

Этот подход имеет две большие проблемы:

- Он заставляет разработчика собирать приложение на старой версии Glibc, чтобы гарантировать его работу на большинстве используемых дистрибутивов Linux, потому что Glibc плох. Это особенно проблематично, если вашему приложению требуется что-то новое, например QT6 или GTK4, которые недоступны в таких старых дистрибутивах.

- Это также означает, что приложение не может работать в системах c Musl.

И будущая стабильность тоже не очень хороша, потому что Glibc иногда все еще нарушает работу пользовательского пространства при обновлениях.

### *Решение*

- ~~Давайте воспользуемся контейнером~~ ❌ Нет, у него куча ограничений и странных особенностей, [очень громоздкий](https://imgur.com/a/appimage-vs-flatpak-size-comparison-QH1dPyb) и зависит от непривилегированных пространств имен пользователей [на которые вы даже не можете полагаться...](https://github.com/linuxmint/mint22-beta/issues/82). Стоит добавить, что в некоторых случаях контейнеры действительно являются единственным жизнеспособным вариантом, особенно для приложений, которые зависят как от 32-битных, так и от 64-битных библиотек, в которых делать это без контейнера будет очень сложно, но да, всегда оставляйте это как крайнюю меру.

- Давайте всё просто скомпилируем статически! Конечно, это работает, скомпилируйте весь Kdenlive статически и сообщите мне, как только закончите.

- Включите в пакет все необходимые приложению библиотеки и не полагайтесь на библиотеку Си-стандартную библиотеку хоста. ✅

Это решение — действительно портативные пакеты приложений, содержащие всё необходимое.

### *Как это работает?*

**Примечание:** В этом разделе объясняются технические детали реализации. Скрипт `quick-sharun` и инструмент `sharun` обрабатывают всё это автоматически, поэтому вам не нужно ничего делать вручную. Это здесь в образовательных целях.

1. Первая проблема, которую нужно преодолеть:

Поскольку мы собираемся включить собственную Си-стандартную библиотеку, это означает, что мы не можем использовать динамический компоновщик хоста, а значит, нам придётся включить собственный `ld-linux/musl.so`, и это создаёт проблему: мы не можем просто внести изменения в бинарные файлы, чтобы использовать встроенный интерпретатор, например, `patchelf --set-interpreter '$ORIGIN/ld-linux.so'`, потому что разрешение `$ORIGIN` выполняется самим интерпретатором.

**Мы можем** - использовать относительный интерпретатор, например, `./ld-linux.so`, но проблема в том, что для этого нам нужно изменить текущий рабочий каталог на это местоположение. Другими словами, для AppImage текущий рабочий каталог изменится на случайную точку монтирования AppImage, и это проблема, если ваше приложение — эмулятор терминала, который, например, открывается в текущем рабочем каталоге.

Вместо этого нам сначала нужно запустить динамический компоновщик, а затем указать ему исполняемый файл, который мы хотим запустить. Это возможно, поэтому наш `AppRun` будет выглядеть следующим образом:

```shell
#!/bin/sh

CURRENTDIR="$(readlink -f "$(dirname "$0")")"

exec "$CURRENTDIR"/ld-linux-x86-64.so.2 "$CURRENTDIR"/bin/app "$@"
```

Однако у этого подхода есть небольшая проблема: `/proc/self/exe` будет `ld-linux-x86-64.so.2` вместо имени запущенного нами исполняемого файла. Для большинства приложений это не проблема, но если она возникает, то довольно серьезная. **Шарун исправляет эту проблему*- (см. ниже), поэтому мы продолжим использовать этот подход для объяснения остальной части кода.

2. Вторая проблема, которую необходимо преодолеть:

Теперь, когда у нас есть собственный динамический компоновщик, Как нам указать приложению использовать все библиотеки, которые находятся в нашем собственном каталоге `lib`?

- ~~LD_LIBRARY_PATH~~ ❌ Нет, ужасная идея. **НИКОГДА НЕ ИСПОЛЬЗУЙТЕ ЭТУ ПЕРЕМЕННУЮ**, она вызывает много проблем, потому что наследуется дочерними процессами, а это значит, что всё, что запускается нашим приложением, будет пытаться использовать наши библиотеки, и это приводит к нездоровому поведению, которое трудно обнаружить. [Например](https://github.com/zen-browser/desktop/issues/2748), эта проблема длилась несколько месяцев, и никто не знал, что происходит, пока я [не удалил](https://github.com/zen-browser/desktop/pull/6156/files) использование `LD_LIBRARY_PATH`, хотя приложению в данном случае даже не нужно было его устанавливать. См. также: [LD_LIBRARY_PATH – или: Как попасть в беду!](https://www.hpc.dtu.dk/?page_id=1180)

- Давайте установим наш rpath как `$ORIGIN/path/to/libs`, это совершенно допустимо! ☑️ Однако во многих случаях это делается не во время компиляции, а с помощью `patchelf` во время выполнения. Хотя в 99% случаев это нормально, в том 1%, когда это что-то ломает, очень сложно обнаружить, что пошло не так.

- Укажем динамическому компоновщику использовать наши встроенные библиотеки напрямую ✅ Это не очень известно, но динамический компоновщик поддерживает флаг `--library-path`, который ведет себя очень похоже на `LD_LIBRARY_PATH`, не являясь переменной, наследуемой другими процессами. Это идеальное решение, которое нам было нужно, поэтому наш пример `AppRun` теперь будет выглядеть так:

```shell
#!/bin/sh

CURRENTDIR="$(readlink -f "$(dirname "$0")")"

exec "$CURRENTDIR"/ld-linux-x86-64.so.2 \
    --library-path "$CURRENTDIR"/lib \
    "$CURRENTDIR"/bin/app "$@"
```

Нам нужно собрать библиотеки и динамический компоновщик, и **почти** всё готово! Однако, чтобы быть полностью готовыми, нам нужно исправить следующие проблемы… **Сборка всех необходимых библиотек не так проста, как просто запустить `ldd` + `cp`**, поэтому нам нужно более надежное решение. Sharun обрабатывает это автоматически (см. ниже).

3. Третья проблема, которую нужно преодолеть:

Давайте сделаем наше приложение перемещаемым. К счастью, это уже возможно практически во всех приложениях. Я часто вижу, как разработчики добавляют исключения в свои приложения, чтобы сделать их переносимыми, **но они редко вообще нужны**, потому что у нас уже есть **спецификация базовых каталогов XDG**, которая очень помогает в этом: <https://specifications.freedesktop.org/basedir-spec/latest/>

Вместо того чтобы жестко прописывать в вашем приложении поиск файлов в `/usr/share`, вам нужно проверять `XDG_DATA_DIRS`, что, скорее всего, уже делает ваше приложение, поскольку большинство библиотек уже следуют этой спецификации.

Затем в нашем `AppRun` мы включаем наш каталог `share` в `XDG_DATA_DIRS`, проблема решена ✅

Аналогичным образом, зависимости, которые мы включаем в пакет, почти всегда имеют средства для обеспечения возможности перемещения любого необходимого им плагина поддержки/файла поддержки, вот лишь несколько примеров:

- `PERLLIB` для Perl

- `GCONV_PATH` для Glibc

- В Qt есть `QT_PLUGIN_PATH`, но также есть другой способ обеспечения возможности перемещения путем создания файла `qt.conf` рядом с исполняемым файлом нашего приложения Qt. **Это намного лучше, потому что эта переменная имеет схожие проблемы с*- `LD_LIBRARY_PATH`

- `PIPEWIRE_MODULE_DIR` и `SPA_PLUGIN_DIR` для PipeWire.

- `VK_DRIVER_FILES` и `__EGL_VENDOR_LIBRARY_DIRS` для MESA (Vulkan и OpenGL) 💪

И многое-многое другое!

Но разве это не слишком много работы — найти и установить все необходимые моему приложению переменные окружения? **Да, это так**

### *Sharun*

Есть решение этой проблемы, созданное [VHSGunzo](https://github.com/vhsgunzo), под названием sharun:

<https://github.com/VHSgunzo/sharun>

- sharun может найти все необходимые вашему приложению библиотеки, **включая те, которые открываются посредством dlopen**. Оказывается, многие приложения зависят от библиотек открытых при помощи dlopen; Это те библиотеки, которые сложно найти с помощью простого `ldd`. Sharun использует скрипт развертывания под названием `lib4bin`, который имеет режим strace; **в этом режиме `lib4bin` открывает приложение с помощью strace, чтобы проверить все библиотеки, загруженную через dlopen, а затем упаковывает её.**

- Sharun также обнаруживает и устанавливает множество [переменных окружения](https://github.com/VHSgunzo/sharun?tab=readme-ov-file#environment-variables-that-are-set-if-sharun-finds-a-directory-or-file), необходимых приложению для работы.

- Он также исправляет проблему с `/proc/self/exe`, который является `ld-linux-x86-64.so.2` 👀 В этом случае он помещает все разделяемые библиотеки и бинарные файлы в `shared/{lib,bin}`, а затем создает жесткую ссылку на каталог `bin` в нашем `AppDir`; Затем, при вызове `bin/app`, автоматически запускается встроенный динамический компоновщик, и исполняемый файл запускается с именем жесткой ссылки, при этом путь к нашим встроенным библиотекам указывается с помощью `--library-path`.

- `sharun` также выполняет функцию `AppRun`, и дополнительные переменные окружения можно добавить, создав рядом с ним файл `.env`, **это означает, что мы больше не зависим от оболочки хоста для запуска нашего приложения.**

- sharun подходит не только для AppImage, его можно использовать везде, где необходимо сделать любое приложение портативным. Вы даже можете создавать псевдостатические бинарные файлы из существующих динамических бинарных файлов, что sharun делает с помощью wrappe.

- sharun также имеет механизмы для исправления приложений, которые не являются локализуемыми, например, WebKitGTK, который жестко запрограммирован на поиск некоторых бинарных файлов в `/usr/lib`, он исправляет это автоматически, внося все необходимые исправления.

Любое приложение, созданное с помощью sharun, в конечном итоге может работать **на любом дистрибутиве Linux**, будь то Ubuntu 14.04, дистрибутивы с библиотекой Musl и даже напрямую в NixOS без каких-либо обёрток (в среде, отличной от FHS).

### *Разве это не слишком громоздко?*

На самом деле нет. Если ваше приложение не использует аппаратное ускорение, включение всех библиотек обычно увеличивает размер приложения менее чем на 6 МБ.

**ОБНОВЛЕНИЕ:** Теперь мы можем собирать Mesa без привязки к LLVM, поэтому AppImage-файлы стали еще меньше, **приложения Qt/GTK с полным аппаратным ускорением могут быть созданы таким образом, чтобы их конечный размер составлял менее 35 МиБ.**

Для приложений с аппаратным ускорением существует проблема, связанная с тем, что Mesa использует `libLLVM.so`, огромную библиотеку размером более 130 МБ, которая используется для множества задач. В дистрибутивах по умолчанию он собирается с поддержкой следующих архитектур:

```shell
AArch64
AMDGPU
ARM
AVR
BPF
Hexagon
Lanai
LoongArch
Mips
MSP430
NVPTX
PowerPC
RISCV
Sparc
SystemZ
VE
WebAssembly
X86
XCore
```

В большинстве приложений LLVM достаточно для поддержки AMDGPU и X86/AArch64.

Мы уже создали такую ​​версию LLVM здесь: <https://github.com/pkgforge-dev/archlinux-pkgs-debloated>, которая уменьшает размер libLLVM.so до 66 МиБ.

Этот пакет и другие раздутые пакеты, которые у нас есть, используются [Goverlay](https://github.com/benjamimgois/goverlay), в результате чего получается AppImage размером **50 МиБ**, который работает на любой системе Linux, что удивительно мало, учитывая, что это приложение включает в себя **Qt** и **Mesa** (Vulkan) среди прочего.

### *А что насчет NVIDIA?*

Nvidia выпускает свой проприетарный драйвер в виде бинарного файла, который уже широко совместим сам по себе. Единственное требование — достаточно новая версия Glibc, что и обеспечивают созданные здесь AppImage, если вы собираете их на дистрибутиве Glibc. Затем вам просто нужно добавить icd-файлы Nvidia в `VK_DRIVER_FILES`, чтобы иметь возможность использовать его без проблем.

Если у вас нет проприетарного драйвера NVIDIA, Mesa уже включает поддержку Nouveau для тех немногих видеокарт, для которых этот драйвер работает (серия NVIDIA GTX 16 или новее).

Само собой разумеется, что Sharun уже сам справляется со всем этим.
